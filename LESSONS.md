# Lessons Learned

## 2026-04-07 - Web Viewer Phase 1

- Streamlit entrega velocidade, mas sem tema explicito o resultado parece utilitario.
- Hierarquia precisa ser definida no topo: contexto da simulacao primeiro, detalhes depois.
- KPI padrao (`st.metric`) e rapido, mas cards customizados trazem melhor percepcao premium.
- Estados vazios e de erro precisam orientar proxima acao; texto tecnico sozinho aumenta atrito.
- Para tabela densa, truncar e oferecer expansao melhora leitura sem perder informacao.
- Reorganizar navegacao (Mapa primeiro) reduz tempo de descoberta para demos.

## 2026-04-07 - Web Viewer Phase 2

- Refinamento visual funciona melhor quando tokens cobrem tipografia, borda, raio e estados.
- Em Streamlit, consistencia perceptiva depende de estilizar tambem controles nativos (tabs/input/button).
- Pequenos ajustes de microcopy reduzem friccao sem mudar comportamento do produto.
- Card KPI com altura minima melhora alinhamento e leitura comparativa.

## 2026-04-07 - Web Viewer Phase 3

- Microinteracao deve ser curta e funcional; quando discreta aumenta percepcao de qualidade.
- Skeleton simples comunica "app vivo" melhor que spinner isolado em telas de conteudo.
- Diferenciar visualmente mensagens de chat melhora leitura sem alterar conversa.
- Polimento visual mais eficiente quando feito sobre base de tokens ja consistente.

## 2026-04-07 - Design QA final

- Foco visivel e requisito de qualidade, nao detalhe opcional.
- Reducao de movimento melhora conforto para parte dos usuarios sem custo funcional.
- Documentacao de design evita regressao de consistencia entre sessoes.
